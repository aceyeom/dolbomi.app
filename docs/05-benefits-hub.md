# 05 · 혜택 허브 (Benefits Hub)

> **상태 박스**
> 우선순위: 빌드 순서 4번.
> 의존성: 기회 엔진([03](./03-opportunity-engine.md))의 "퀘스트로 만들기" 핸드오프, 카탈로그([04](./04-catalog/README.md)), 정직성([10](./10-honesty-notes.md)).
> 미해결 질문: §5.5 참조.

"군인이라서 받는 모든 것." 혜택 허브는 *정적 카탈로그* 가 아니라 **혜택 → 기회 → 퀘스트 경로** 의 진입점이다. 보여주기만 하면 v3처럼 죽는다 — 핵심은 한 탭으로 **청구 경로(claim-path)** 를 만드는 것.

---

## 5.0 약점 / 리스크 (먼저 솔직하게)

1. **정적 목록은 이미 검색되는 정보다.** 혜택을 나열만 하면 나라사랑포털·블로그와 차별 0. v3가 여기서 죽었다. → 해결: 허브의 가치는 *나열*이 아니라 **"→ 퀘스트로 만들기"**(§5.1) — 추상적 혜택을 날짜 박힌 개인 경로로 바꾸는 것.
2. **자격 판정에 민감정보가 필요해 보인다.** 부모 소득·카드 데이터로 필터링하면 정확하지만, **그게 v3의 신뢰 함정**이었다. → 해결: 자격은 **클라이언트 측 셀프 필터**, 아무것도 저장 안 함(§5.1). 정확도를 약간 포기하고 신뢰를 산다.
3. **혜택 정보가 늙는다(stale).** 응시료·매칭율·학점 한도는 매년 바뀐다. 틀린 숫자 = 신뢰 즉사(이 코호트는 간파함). → 해결: 모든 항목 `source_url` + `verified_at`, **90일 경과 시 ⚠️ stale 배지**(§5.3, → [03](./03-opportunity-engine.md) §3.1).
4. **군별×계급 자격 매트릭스가 복잡하다.** "이 혜택 나도 되나?"가 군별·계급·복무형태로 갈림. 틀리게 "된다"고 하면 헛걸음, "안 된다"고 하면 기회 박탈. → 해결: 셀프 필터는 **보수적 표시 + "부대/포털에서 최종 확인"** 명시. 단정 금지(§5.2).
5. **정부 돈 vs 리퍼럴 충돌.** 허브가 제휴 상품을 정부 혜택보다 위에 놓으면 신뢰 붕괴. → 해결: **정부 돈 먼저**(자기개발비·무료검정), 리퍼럴은 그 다음(→ [06](./06-referral-engine.md) §6.5). 허브 정렬은 *사용자 이득순*, 매출순 아님.

---

## 5.1 메커닉 — 셀프 필터, 아무것도 저장 안 함

사용자가 필터를 고른다(**군별 / 계급 / 관심분야**). 자격은 **클라이언트 측에서 계산**한다. 우리는 의도적으로 **수집하지 않는다** — 부모 소득, 카드 데이터, 민감 정보 일체(이게 v3의 함정이었다). 같은 가치, 민감 데이터 저장 0, 신뢰 비용 0.

```
filterBenefits(catalog, {군별, 계급?, 관심분야?}):
  return catalog.filter(b =>
     matches(b.eligibility.군별, 군별) &&
     (!b.eligibility.계급 || rankOK(b.eligibility.계급, 계급)) )
  // 필터 입력은 메모리에만; 영속화 0 (→ 10 정직성)
```

각 카드 → **"→ 퀘스트로 만들기"** 버튼이 그 혜택을 기회 엔진(→ [03](./03-opportunity-engine.md))에 넘겨 **청구 경로(claim-path)** 를 생성. 핸드오프 페이로드:

```jsonc
{ opportunity_id,            // 혜택은 카탈로그의 Opportunity로 모델링
  intent: "claim",           // (vs 일반 "achieve")
  source: "benefits-hub" }
```

엔진은 이걸 받아 "신청 → 서류 → 제출 → 확인"식 마일스톤 경로를 만든다(사실은 카탈로그에서 주입, → [03](./03-opportunity-engine.md) §3.4).

## 5.2 자격 표시 규약 (단정 금지)

- 셀프 필터 결과는 **"해당 가능성 있음"**, 단정 아님.
- 각 카드에 **"최종 자격은 [정확한 포털/부대]에서 확인"** 1줄 고정.
- 군별/계급 불명확 항목은 **보수적으로 표시**(놓침 < 헛걸음). 단, 완전 배제도 안 함 — "조건부" 섹션으로.

## 5.3 신선도(stale) 메커닉

```
isStale(b) = daysSince(b.verified_at) > 90
```
- stale → 카드에 **⚠️ 배지 + "마지막 확인 YYYY-MM"**.
- stale 항목은 "퀘스트로 만들기" 가능하되, 엔진이 경로에 **"먼저 최신 정보 확인" 첫 서브퀘스트**를 강제 삽입.
- 전 항목 `source_url` 보유 → 탭하면 1차 출처로(→ [docs/appendix-sources.md](./appendix-sources.md)).

## 5.4 혜택 카탈로그 (실제)

> 모든 수치는 **시드값** — 각 항목 `source_url` + `verified_at` 보유. 카탈로그 본문은 [04](./04-catalog/README.md), 검증은 money/craft 등 카테고리 파일 참조.

| 혜택 | 받는 것 | 청구처 | 1차 스탯 |
|---|---|---|---|
| 장병내일준비적금 + 매칭 | 정부 매칭 → 전역 시 목돈 | 은행 앱 + 나라사랑카드 | Money |
| 자기개발비 | 응시료·도서·강좌·용품 환급(연 한도) | 나라사랑포털 → 자기개발 | Craft |
| 국가기술자격 무료검정 | 다수 종목 무료 시험 | 국방부 위탁검정 | Craft |
| 대학 원격강좌(학점) | 학기/연 한도 학점, 환급 | 나라사랑포털 → 군 e-러닝 | Craft |
| 군 e-러닝 강좌 | 어학·IT·자격증·검정고시 무료/할인 + 전화영어 | 나라사랑포털 | Craft/Mind |
| 어학 응시료 할인 | 할인된 TOEIC 등 | 나라사랑포털 제휴 | Craft |
| 나라사랑카드 혜택 | 할인/포인트 | 카드 혜택 | Money |
| 제대군인·청년 지원 | 취업·정책(전역 후) | 셀프 필터 → 포털 | Edge/Money |

*(구체 수치·일정·자격은 카테고리 파일에서 검증값으로 관리. 허브는 이 인덱스 + 청구 핸드오프 + stale 플래그를 담당.)*

## 5.5 미해결 질문 (Phase 2/실데이터)

- 군별×계급 자격 매트릭스의 데이터 소스·갱신 주기.
- stale 임계(90일)와 항목별 변동 주기 차이(연 1회 vs 상시).
- "퀘스트로 만들기" 청구 경로의 템플릿 골격(혜택 유형별).
- 셀프 필터 "조건부" 표시가 헛걸음 vs 기회박탈 사이에서 맞는 균형인지.
